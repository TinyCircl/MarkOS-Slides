# PPTX Export Architecture

This document describes the current editable `pptx` export architecture in MarkOS.

MarkOS now exports `pptx` through the real web render pipeline instead of through a separate layout engine.

In short:

`Markdown -> build/export HTML -> headless browser measurement -> export model -> pptxgenjs`

## Scope

This document is about the local OSS implementation in `MarkOS-Slides`.

It covers:

- local CLI export
- the `markos-web` engine export path
- the browser-side DOM collector
- the native PPTX writer

It does not describe the service adapter in `MarkOS-Slides-renderer`.
That repository still has its own product boundary and may reject `pptx` at the API layer even though local CLI export now supports it.

## Current User Surface

The current user entrypoint is:

```bash
markos export <deck> --format pptx
```

Equivalent repo-root command:

```bash
npm run markos:export -- examples/tokyo3days --format pptx
```

The command is implemented in:

- [packages/cli/src/index.mjs](/Users/xuao/Desktop/MarkOS/MarkOS-Slides/packages/cli/src/index.mjs)

## High-Level Flow

The current PPTX export flow is:

1. CLI resolves the deck, output directory, work directory, and format.
2. CLI builds a normalized local project input and injects the bundled theme source.
3. Core calls the `markos-web` engine with `format: "pptx"`.
4. The engine builds a temporary static site and starts a temporary local manifest server.
5. A headless browser opens the real `/export/` page for that build.
6. The browser-side runtime walks the final DOM and collects exportable nodes.
7. Node.js receives that export model and passes it to `pptxgenjs`.
8. `pptxgenjs` writes a native `.pptx` file with editable text, shapes, and images.
9. MarkOS cleans up the temporary site and work directory.

## CLI Layer

The CLI accepts `pdf` and `pptx` in the same command surface.

Current behavior:

- normalizes `--format`
- rejects unsupported formats early
- builds the same local project input used by `markos build`
- preserves theme CSS and deck-local CSS
- chooses the output file name and suffix

The CLI does not contain PPTX layout logic.
It only prepares input and delegates artifact generation to core.

## Core Export Orchestration

The `markos-web` engine owns the actual PPTX export flow.

Current responsibilities:

- build a temporary static site for the deck
- write a preview manifest
- start a temporary local site server
- launch a headless browser against the real export page
- collect a browser-measured export model
- write the `.pptx`
- clean up temporary files

This happens in:

- [packages/core/src/engines/markos-web/index.mjs](/Users/xuao/Desktop/MarkOS/MarkOS-Slides/packages/core/src/engines/markos-web/index.mjs)

Important current helpers:

- `prepareExportSite(...)`
- `collectExportModelFromBrowser(...)`
- `renderPptxArtifact(...)`
- `exportArtifact(...)`

## Browser Choice

PPTX export uses a local Chrome or Chromium executable through `playwright-core`.

Current browser resolution order supports:

- `MARKOS_EXPORT_BROWSER`
- `MARKOS_PPTX_BROWSER`
- `MARKOS_PDF_BROWSER`
- `MARKOS_CHROME_PATH`
- `CHROME_PATH`

`MARKOS_EXPORT_BROWSER` is the preferred override because it works for both `pdf` and `pptx`.

## Export Page Source Of Truth

MarkOS does not measure arbitrary author HTML.

It measures the real MarkOS export page generated by:

- [packages/core/src/engines/markos-web/render.mjs](/Users/xuao/Desktop/MarkOS/MarkOS-Slides/packages/core/src/engines/markos-web/render.mjs)

That matters because:

- the deck has already been parsed
- the theme has already been applied
- deck-local CSS has already been bundled
- the browser sees the final layout that users actually authored

The collector runs only on:

```text
/export/?collect=1
```

## DOM Contract

The current implementation uses a lightweight DOM contract.

It does not build a separate IR layer first.
Instead, it marks important DOM regions and measures them after layout is final.

Current export-relevant attributes include:

- `data-markos-role="slide"`
- `data-markos-template`
- `data-markos-slide-index`
- `data-markos-slide-title`
- `data-markos-role="slide-layout"`
- `data-markos-role="header"`
- `data-markos-role="column-left"`
- `data-markos-role="column-right"`

This contract is currently applied by the renderer for:

- cover layouts
- default layouts
- two-column layouts
- export slide wrappers

## Browser-Side Collector

The collector lives inside the runtime script generated by:

- [packages/core/src/engines/markos-web/render.mjs](/Users/xuao/Desktop/MarkOS/MarkOS-Slides/packages/core/src/engines/markos-web/render.mjs)

The collector currently does these jobs:

1. Wait for fonts and images.
2. Find all export slide elements.
3. Measure the slide viewport.
4. Collect:
   - shape nodes
   - image nodes
   - text nodes
5. Read background color from the effective slide layout node.
6. Serialize the export model into `window.__MARKOS_EXPORT_MODEL__`.

Current helper groups in the runtime:

- geometry helpers
- color and border helpers
- shape detection
- text extraction
- image extraction
- export model assembly

## Measurement Rules

The collector uses browser pixels from `getBoundingClientRect()`.

Current rules:

- geometry is measured relative to each `.slide-page`
- invisible elements are skipped
- `img` nodes are exported separately from panel/background shapes
- text is collected from headings, paragraphs, lists, table cells, captions, and preformatted blocks
- list text is flattened into bullet-prefixed lines
- simple one-sided borders can become thin exported bars

The current collector is intentionally pragmatic.
It tries to preserve the shapes and text that matter most for editing, not every CSS detail.

## Coordinate Conversion

The PPTX writer converts browser pixels to PowerPoint dimensions using:

- `96 px = 1 inch`
- `4/3 px = 1 point`

That conversion happens in:

- [packages/core/src/engines/markos-web/pptx.mjs](/Users/xuao/Desktop/MarkOS/MarkOS-Slides/packages/core/src/engines/markos-web/pptx.mjs)

Current helpers:

- `pxToInches(...)`
- `pxToPoints(...)`

## PPTX Writer

The native writer uses:

- `pptxgenjs`

Current responsibilities:

- define a custom PPTX layout that matches the measured deck viewport
- create one PowerPoint slide per measured MarkOS slide
- apply slide background color
- sort nodes by layer and order
- render shapes
- render images
- render text

Current mapping:

`shape`
- PPTX rectangle
- PPTX rounded rectangle
- PPTX ellipse

`image`
- PPTX image object

`text`
- PPTX text box

Current text mapping preserves:

- font family
- font size
- weight
- italic
- alignment
- color

That is why the resulting file is editable instead of a page screenshot.

## What Is Editable Today

Today the generated PPTX keeps these objects as native PowerPoint objects when possible:

- headings
- paragraph text
- list text
- simple colored panels
- border-derived accent bars
- images

This already gives a practical editable result for current themes.

## What Is Not Fully Modeled Yet

The current implementation is intentionally light and has limits.

Examples of current limits:

- no dedicated rich-text run model inside a single text box
- no dedicated native table writer yet
- no explicit SVG path export
- no semantic chart model
- no full CSS effect parity
- no browser-independent font metrics

So the current system is:

- native where practical
- approximate where acceptable
- not a universal HTML-to-PPTX converter

## Why This Is Still Not A Screenshot Exporter

Even though the browser is used for layout measurement, the final `.pptx` is not just a stack of screenshots.

The browser is used to answer:

- where things ended up
- how large they are
- how they are styled

Then MarkOS rebuilds those elements as editable PowerPoint objects.

This keeps the implementation lightweight without giving up editability.

## Current Boundaries

This document describes the local OSS export architecture only.

It does not mean every surrounding integration already supports the same surface.

In particular:

- local MarkOS CLI now supports `pptx`
- the service adapter repository may still restrict `pptx` until its own runtime and dependency boundary is updated

## Summary

The current MarkOS PPTX architecture is:

- HTML-first
- browser-measured
- DOM-contract-driven
- native-object-oriented at export time
- intentionally lightweight

It avoids a heavy IR layer while still producing editable PowerPoint files from the real rendered slide surface.
